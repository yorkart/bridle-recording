# Bridle Recording Direction

## 核心目标

Bridle Recording 的长期目标是围绕模型交互建立一套可观测、可剪裁、可复现的测试集与 mock 服务，同时保持在线代理链路的完全透明性。

## 基本分层

1. 在线代理链路只负责透明转发。
2. 在线录制链路只负责 100% 原样保存 recorder 观测到的原始流量。
3. 观测页面基于原始录制派生结构化视图，用于浏览模型交互过程。
4. 剪裁流程基于原始录制生成可提交的测试集内容。
5. mock 服务只基于剪裁后的测试集提供响应。

## 原始录制约束

原始录制是事实来源，必须保存请求头、请求体、响应头、响应体、SSE 事件、WebSocket 帧等 recorder 观测到的原始内容。在线录制阶段不得做脱敏、替换、canonicalization、兼容转换、tools/skills 移除或其他派生处理。

## 观测页面方向

页面应展示模型交互过程的派生视图，包括但不限于请求/响应时间线、user input、tools 列表、tool call 参数与返回值、skills/apps/plugins 信息、meta 信息、SSE 事件和 WebSocket 帧。派生视图不能替代原始录制，也不能反向影响代理与录制链路。

## 剪裁与测试集

用户可以在页面上对已录制内容进行剪裁，生成可提交到当前 git 仓库的测试集。剪裁内容可以包含敏感词处理、移除部分或全部 tools、移除部分或全部 skills、移除 prompt 中的其他派生片段，以及保留测试所需的请求/响应序列。

测试集应记录来源信息和剪裁策略，使剪裁内容可审查、可复现、可追溯。默认提交对象应是剪裁后的测试集，而不是未剪裁的原始录制。

## Mock 服务匹配

mock 服务从 request body 中提取 user input，用它定位对应的剪裁 session。定位 session 之后，mock 服务必须对当前 request 按该 session 的预期顺序进行严格匹配校验。只有匹配成功，才返回对应响应；否则必须报错。

user input 只用于定位 session，不能用于放宽匹配。mock 服务不得隐式回退到原始录制。

## 架构边界

任何提高可读性、可检索性、安全性、兼容性或 mock 便利性的能力，都应属于原始录制之后的派生流程。它们不能进入在线代理主链路，也不能改变在线原始录制的内容。
